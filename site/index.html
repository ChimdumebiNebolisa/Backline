02 / One run at a time

The change is easier to review when it leaves a trail.

Backline turns a YAML check into durable evidence. The CLI submits. The worker executes. PostgreSQL keeps the record. The next run can answer what changed.

04 / Policy at the edge

Make regressions loud at the boundary.

Run policy against stored results and diff. Keep terminal output for people. Emit JUnit when the CI system needs a machine-readable failure.

Read the CI guide ->
backline.ymlpolicy
policy:
  max_newly_failing: 0
  max_errored_checks: 0
  max_latency_regression_ms: 200

$ backline run \
  --enforce-policy \
  --junit-output ./build/policy.xml

06 / Honest limits

Useful because it stays honest.

Not a Postman replacement

Use an interactive client for exploration. Use Backline when behavior needs repeatable history.

No hosted monitoring

There is no SaaS dashboard or cloud sync workflow. Run the stack locally or in CI.

Bounded evidence

Response previews are bounded and sensitive headers are redacted according to the repository guardrails.

Intentional sample failures

The sample API includes failing and slow checks so the demo can show a real regression outcome.
