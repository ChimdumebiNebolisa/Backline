/ the workflow

One run. Every signal.

Backline makes the full path visible: configuration becomes a queued run, a worker records results, and the next run has something real to compare.

  1. 01

    Configure

    Describe checks in backline.yml with expected status, latency limits, and assertions.

    YAML
  2. 02

    Queue

    The CLI syncs checks and asks the API to create a durable run.

    QUEUED
  3. 03

    Execute

    The worker claims one queued run, calls the target API, and writes bounded results.

    RUNNING
  4. 04

    Compare

    History and diff endpoints show new failures, passes, latency changes, and missing checks.

    PASSED / FAILED
  5. 05

    Report

    Render Markdown or JSON from API data for a review artifact or CI consumer.

    REPORT

/ the difference

A request is a moment. A ledger is a trail.

Backline is for repeatable checks whose results should still be there tomorrow. Each run keeps status, timing, config identity, events, results, and a comparison path.

/ policy in CI

Make regressions loud at the boundary.

Run policy against the stored results and diff. Keep the terminal output for humans; emit JUnit when the CI system needs a machine-readable failure.

/ boundaries

Useful because it stays honest.

Backline is intentionally local-first. These constraints are part of the product, not footnotes hidden behind a glossy interface.

—

Not a Postman replacement

Use an interactive client for exploration. Use Backline when behavior needs repeatable history.

—

No hosted monitoring

There is no SaaS dashboard or cloud sync workflow. Run the API, worker, database, and sample locally.

—

Bounded evidence

Response previews and sensitive headers are bounded or redacted according to the repository guardrails.

—

Intentional sample failures

The sample API includes failing and slow checks so the demo can show a real regression outcome.
